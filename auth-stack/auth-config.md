# Auth config

The `betterAuth()` instance, its plugins, additional fields, database hooks, and the Zod-validated server env it reads from. This file is the canonical home for those topics — the rest of the skill references back here.

## Dependencies

```jsonc
// package.json (subset that matters)
{
  "dependencies": {
    "@better-auth/prisma-adapter": "^1.5.4",
    "better-auth":                 "^1.5.4",

    "@tanstack/react-start":       "^1.166.0",
```

`better-auth` and `@better-auth/prisma-adapter` versions track each other — bump them together. Zod v4 is assumed for `z.email()` / `z.url()` standalone schemas; on Zod v3 substitute `z.string().email()` / `z.string().url()`.

## The `auth` instance

```ts
// <server>/lib/auth.ts
import { prismaAdapter } from "@better-auth/prisma-adapter";
import { betterAuth } from "better-auth";
import { magicLink } from "better-auth/plugins";
import { tanstackStartCookies } from "better-auth/tanstack-start";

import { sendMagicLinkEmail } from "<server>/lib/email/senders";
import { serverEnv } from "<server>/lib/env.server";
import { prisma } from "<server>/lib/prisma";

function getTrustedOrigins(baseUrl: string): Array<string> {
  const raw = serverEnv.BETTER_AUTH_TRUSTED_ORIGINS;
  if (!raw) return [baseUrl];
  return raw.split(",").map((origin) => origin.trim()).filter(Boolean);
}

export const auth = betterAuth({
  database: prismaAdapter(prisma, { provider: "mongodb" }),
  user: {
    additionalFields: {
      role:            { type: "string", required: false, input: false, defaultValue: "user" },
      preferredLocale: { type: "string", required: false, input: false, defaultValue: "en" }
    }
  },
  databaseHooks: {
    user: {
      create: {
        before: (user) =>
          Promise.resolve({
            data: {
              ...user,
              role: typeof user.role === "string" ? user.role : "user"
            }
          })
      }
    }
  },
  advanced: { database: { generateId: false } },
  baseURL:        serverEnv.BETTER_AUTH_URL,
  basePath:       "/api/auth",
  secret:         serverEnv.BETTER_AUTH_SECRET,
  trustedOrigins: getTrustedOrigins(serverEnv.BETTER_AUTH_URL),
  plugins: [
    tanstackStartCookies(),
    magicLink({
      expiresIn: 60 * 15,
      storeToken: "hashed",
      sendMagicLink: async ({ email, url }) => {
        await sendMagicLinkEmail({ email, url });
      }
    })
  ]
});
```

## Key choices and why

### `prismaAdapter(prisma, { provider: "mongodb" })`

The Prisma adapter knows about Mongo's quirks — notably the lack of foreign-key support, which the schema handles via `relationMode = "prisma"`. Don't substitute the generic adapter or pass a different `provider` — the adapter uses this to pick the correct ID strategy.

### `advanced.database.generateId: false`

Mongo `ObjectId` values are generated by Prisma via `@default(auto()) @db.ObjectId`. With `generateId: true` (the default), Better Auth would try to insert string IDs and break the round-trip. **This setting is non-negotiable when using the Prisma + Mongo adapter.**

### `additionalFields` semantics — the canonical reference

```ts
{
  type: "string",
  required: false,
  input: false,        // ← cannot be set via the public sign-up payload
  defaultValue: "user" // ← honored by Better Auth at create time (with caveat below)
}
```

`input: false` is the **security boundary**: a sign-up POST body cannot promote itself to `admin`. Always set this on any field that grants permission, trust, or capability.

`required: false` keeps the sign-up flow flexible — Better Auth doesn't reject sign-ups that don't provide a value.

The `defaultValue` is **not** a guarantee under the Prisma adapter. Better Auth's `defaultValue` and Prisma's `@default(...)` interact unpredictably for additional fields (the adapter sometimes inserts a row before the default is applied). The `databaseHooks.user.create.before` hook is the belt-and-braces enforcement. If you add an additional field with a default that gates behavior, **mirror it in the hook**.

### `databaseHooks.user.create.before`

Runs synchronously before the Prisma `create` call. Returns `{ data }` — the full record to insert. Use it to:

- Force defaults that `additionalFields.defaultValue` doesn't reliably set under the Prisma adapter (the common case).
- Normalize input (e.g. lowercase email) — though the magic-link plugin already lowercases emails.
- Reject creation conditionally (throw — it propagates as an auth error).

**Don't read across collections here.** It runs in the hot path of every sign-up; a relational lookup turns sign-up into a multi-second operation.

### `magicLink` plugin options

| Option              | Value          | Why                                                                  |
| ------------------- | -------------- | -------------------------------------------------------------------- |
| `expiresIn`         | `60 * 15`      | 15 min. Long enough for inbox delays, short enough to limit replay.  |
| `storeToken`        | `"hashed"`     | DB stores `sha256(token)`; raw token only exists in the URL.         |
| `sendMagicLink`     | calls SES      | Hands off to your email layer — keeps email composition out of auth. |

The token in the email URL is single-use. Better Auth deletes the `Verification` row on first verify — there is no replay window.

### `tanstackStartCookies()`

Required for Better Auth to set/read its session cookie inside TanStack Start's request lifecycle. Without it, `auth.api.getSession({ headers })` silently returns `null` in SSR. If you migrate to a different framework (Next.js, Remix, Hono), swap in that framework's Better Auth cookie plugin — same role, different shim.

### `trustedOrigins`

Comma-separated CSRF-protection allowlist. Every host that can call `/api/auth/*` (production, preview, notification subdomains) must appear. The `getTrustedOrigins` helper falls back to just `[baseURL]` if `BETTER_AUTH_TRUSTED_ORIGINS` is unset — fine for local dev, but in production the env var should be required.

### `basePath: "/api/auth"`

Must match the catch-all route file location (`<routes>/api/auth/$.ts`). If you move the catch-all, update both — they're a contract.

## Server env (`<server>/lib/env.server.ts`)

```ts
import { z } from "zod";

const serverEnvSchema = z.object({
  NODE_ENV: z.enum(["development", "test", "production"]).default("development"),

  DATABASE_URL: z.string().min(1),

  BETTER_AUTH_SECRET:          z.string().min(1),
  BETTER_AUTH_URL:             z.url(),
  BETTER_AUTH_TRUSTED_ORIGINS: z.string().min(1),

  AWS_BUCKET_NAME:    z.string().min(1),
  AWS_REGION:         z.string().min(1),
  AWS_CLOUDFRONT_URL: z.string().min(1),
  // Optional → SDK falls back to default credential chain (IAM role)
  AWS_ACCESS_KEY_ID:     z.string().optional(),
  AWS_SECRET_ACCESS_KEY: z.string().optional(),

  SES_FROM_EMAIL:           z.email(),
  SES_MARKETING_FROM_EMAIL: z.email().optional(),

  PUBLIC_APP_URL: z.url().default("http://localhost:3000")

  // …project-specific keys (Stripe, AI providers, Redis, etc.)
});

type ServerEnv = z.infer<typeof serverEnvSchema>;

function isTestMode(): boolean {
  return process.env.NODE_ENV === "test" || process.env.VITEST === "true";
}

export const serverEnv: ServerEnv = isTestMode()
  ? (process.env as unknown as ServerEnv)
  : serverEnvSchema.parse(process.env);
```

### Why test mode skips validation

Test suites (Vitest, etc.) typically mock `serverEnv` per test or stub individual modules — forcing the full schema to parse would require every test to populate every env var. `isTestMode()` returns `true` for `NODE_ENV=test` or `VITEST=true`, and the bypass is intentional. **Dev and prod still fail fast** — that's where validation matters.

If you don't use Vitest, replace the second branch of `isTestMode()` with whatever sentinel your test runner sets.

### Why AWS credentials are optional

In production, the IAM role on the instance grants S3/SES access. Setting `AWS_ACCESS_KEY_ID` would shadow the IAM credentials and create a dual-source failure mode (rotation drift, leaked-key blast radius). Local dev sets them explicitly. Both `getSesClient()` and `getS3Client()` apply the fallback identically — see [email-ses.md](email-ses.md) and [storage-s3.md](storage-s3.md).

### Zod v4 vs v3

The schema uses Zod v4's standalone `z.email()` and `z.url()`. On Zod v3, substitute `z.string().email()` and `z.string().url()`. Don't mix — pick one form per file.

## Hard rules

- **Never** instantiate a second `betterAuth(...)` — there is one and only one auth instance.
- **Never** flip `generateId` to `true` without simultaneously removing all `@db.ObjectId` decorators (which would be a much larger schema migration).
- **Never** lower `expiresIn` below ~5 minutes (real users hit inbox delays) or raise it past ~30 minutes (replay window).
- **Never** change `storeToken` from `"hashed"` — the only legitimate alternatives leak raw tokens.
- **Never** add an additional field via Better Auth without mirroring it in `prisma/schema.prisma`. The adapter does **not** auto-migrate.
- **Never** add an additional field with a default that gates behavior without mirroring the default in `databaseHooks.user.create.before` — `additionalFields.defaultValue` is unreliable under the Prisma adapter.
- **Never** read `process.env.X` outside `env.server.ts`. Add the key to the Zod schema and import `serverEnv` instead.
- **Never** read `.env` / `.env.local` from agent tools — secrets. Read `.env.example` and ask the user.
